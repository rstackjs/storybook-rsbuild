# Next.js

Storybook for Next.js & Rsbuild bridges Storybook's preview runtime to Next.js's own build pipeline via `next-rspack`, letting you develop Next.js components in isolation while reusing the same loaders, SWC options, and `next.config.ts` settings as your application.

The framework adds **no build-configuration language of its own**: you keep configuring your app through `next.config.ts`, and the bridge reuses it as-is. See [Configuration mental model](#configuration-mental-model) for the one rule that decides where any given setting belongs.

:::warning Experimental
`storybook-next-rsbuild` relies on Next.js internals and moves with the Next.js release cadence. Any `next` minor or patch upgrade may break compatibility — pin `next` for reproducibility and upgrade `storybook-next-rsbuild` together with `next`.
:::

## Version matrix

`@rsbuild/core` is a peer dependency: you install the version whose transitive `@rspack/core` matches the one `next-rspack` pulls in. If the two sides ship different `@rspack/core` copies, Next.js's extracted plugins bind to a different `NormalModule` class than the one Storybook's compilation actually uses — the framework throws at startup when it detects this.

Pick the row that matches your `next` version and install the listed `@rsbuild/core`:

<div style={{ width: '100%', maxWidth: '700px' }}>

| `next` | `next-rspack` | `@rspack/core` (transitive) | `@rsbuild/core` |
| :--- | :--- | :--- | :--- |
| `15.3.x` | same as `next` | `1.3.4` | `1.3.6` |
| `15.4.x` | same as `next` | `1.4.5` | `1.4.5` |
| `15.5.x` | same as `next` | `1.4.5` | `1.4.5` |
| `16.0.x` | same as `next` | `1.5.0` | `1.5.0` or `1.5.1` |
| `16.1.x` | same as `next` | `1.6.7` | `1.6.14` |
| `16.2.x` | same as `next` | `1.6.7` | `1.6.14` |

</div>

Notes:

- `next-rspack` must be installed at the exact same version as `next` (single release train, no cross-version compatibility).
- `@rsbuild/core` pins `@rspack/core` to an exact version (not a range), so only the specific `@rsbuild/core` versions listed above resolve to the target `@rspack/core`. Installing a newer `@rsbuild/core` within the same minor generally shifts the `@rspack/core` pin and breaks the invariant.
- From Next 16 onwards, the chain is `next-rspack → @next/rspack-core → @rspack/core`. For Next 15, `next-rspack` depends on `@rspack/core` directly.
- If the framework aborts with an `@rspack/core` mismatch error, compare the two paths it prints: they should be the same file. Realigning via this table (or via pnpm `overrides` / yarn `resolutions` forcing `@rspack/core` to the target version) is the supported fix.

## Getting started

### Installation

Install `storybook-next-rsbuild` and `next-rspack` together. Next.js internally calls `require('next-rspack/rspack-core')` while building, so `next-rspack` must be resolvable from your project — a missing peer will surface as an opaque module-not-found error at Storybook start.

<PackageManagerTabs command="install storybook-next-rsbuild next-rspack -D" />

Keep `next-rspack` on the same version as `next`.

### Configure `.storybook/main.ts`

```ts twoslash title=".storybook/main.ts"
// @noErrors
import type { StorybookConfig } from 'storybook-next-rsbuild'

const config: StorybookConfig = {
  framework: 'storybook-next-rsbuild',
  stories: ['../src/**/*.stories.@(js|jsx|mjs|ts|tsx)'],
  addons: ['@storybook/addon-docs'],
}

export default config
```

That's it — the framework will auto-detect `next.config.{js,ts,mjs}` at the project root. If your config lives elsewhere, set `nextConfigPath`:

```ts twoslash title=".storybook/main.ts"
// @noErrors
import type { StorybookConfig } from 'storybook-next-rsbuild'

const config: StorybookConfig = {
  framework: {
    name: 'storybook-next-rsbuild',
    options: {
      nextConfigPath: '../next.config.ts',
    },
  },
  stories: ['../src/**/*.stories.@(js|jsx|mjs|ts|tsx)'],
}

export default config
```

Relative paths resolve against the Storybook config directory (`.storybook`).

## Configuration mental model

`storybook-next-rsbuild` loads your `next.config.{js,ts}` with Next.js's own config loader and asks `next-rspack` to emit the full build configuration, then merges the resulting aliases, loaders, defines, and plugins into Rsbuild. The practical consequence is the single most important thing to internalize:

> **Whatever Next.js compiles → configure it in `next.config.ts` (Next.js style). Whatever belongs to the Storybook preview build itself → configure it in `.storybook/main.ts` (Rsbuild style).**

There is no third, framework-specific config surface to learn or keep in sync. The settings that make `next dev` work already make Storybook work, because the same `next.config.ts` drives both.

### Who owns what

| Concern | Owner | Configure in | Style |
| :--- | :--- | :--- | :--- |
| `'use client'` / `server-only`, JSX runtime, SWC transforms | Next.js | `next.config.ts` (automatic) | Next.js |
| `transpilePackages`, `optimizePackageImports` | Next.js | `next.config.ts` | Next.js |
| `compiler.*` (e.g. `styledComponents`, `emotion`) | Next.js | `next.config.ts` | Next.js |
| Module `resolve.alias` / `fallback`, custom loaders | Next.js | `next.config.ts` → `webpack()` | Next.js |
| `next/font`, `next/image` | Framework (bridged) | `next.config.ts` (automatic) | Next.js |
| CSS Modules, global CSS, PostCSS / Tailwind | Rsbuild | works out of the box | — |
| Sass / Less preprocessors | Rsbuild | `.storybook/main.ts` → `rsbuildFinal` | Rsbuild |
| Storybook-preview-only build tweaks | Storybook builder | `.storybook/main.ts` → `rsbuildFinal` / `webpackFinal` | Rsbuild / webpack |

The split follows the bridge's design — each tool keeps what it does best: **Next.js owns JS/TS compilation and module resolution; Rsbuild owns the CSS pipeline; Storybook owns the preview.** The full rationale (the "four-party contract" between Next.js, Rspack, Rsbuild, and Storybook) lives in [`packages/framework-next/AGENTS.md`](https://github.com/rstackjs/storybook-rsbuild/blob/main/packages/framework-next/AGENTS.md).

### Deciding where a setting goes

1. **Would you put it in `next.config.ts` to make `next dev` / `next build` work?** Leave it there — it's bridged automatically. Don't duplicate it into `.storybook`.
2. **Is it a CSS preprocessor that needs a plugin (Sass, Less, Stylus)?** Add the matching Rsbuild plugin via `rsbuildFinal`. Rsbuild — not Next.js — runs the CSS pipeline in Storybook, so you extend it the Rsbuild way. (CSS Modules, plain CSS, and PostCSS/Tailwind need nothing.)
3. **Is it a tweak only the Storybook preview needs** (an alias just for stories, a rule mutation)? Use `rsbuildFinal` (Rsbuild style) or `webpackFinal` (webpack style).

### What gets bridged from `next.config.webpack()`

If your project customizes webpack in `next.config.ts`, the bridge forwards your **rules, aliases, fallbacks, and externals** automatically. **Plugins are gated** behind the `forwardNextConfigPlugins` option (default `false`): most plugins people add target Next.js's production pipeline (build-manifest writers, source-map uploaders, stats emitters) and either no-op in Storybook or crash rspack's IPC channel (e.g. `copy-webpack-plugin`'s `processAssets` hook). Opt in only for a client-side plugin you've verified against `@rspack/core`:

```ts twoslash title=".storybook/main.ts"
// @noErrors
import type { StorybookConfig } from 'storybook-next-rsbuild'

const config: StorybookConfig = {
  framework: {
    name: 'storybook-next-rsbuild',
    options: {
      // Forward DefinePlugin / client-side plugins from next.config.webpack().
      forwardNextConfigPlugins: true,
    },
  },
  stories: ['../src/**/*.stories.@(js|jsx|mjs|ts|tsx)'],
}

export default config
```

## Framework options

<div style={{ width: '100%', maxWidth: '720px' }}>

| Option | Type | Default | Description |
| :--- | :--- | :--- | :--- |
| `nextConfigPath` | `string` | auto-detected | Absolute or relative path to a `next.config.{js,ts,mjs}` file. |
| `forwardNextConfigPlugins` | `boolean` | `false` | Forward plugins added by `next.config.webpack()` into Storybook's bundler. Rules, aliases, fallbacks, and externals are always forwarded — only plugins are gated. |
| `image` | `object` | `{}` | Accepted for compatibility with `@storybook/nextjs`; the runtime `next/image` mock currently falls through to upstream defaults. |
| `builder` | `BuilderOptions` | `{}` | Options forwarded to `storybook-builder-rsbuild`. See the [Configuration](/guide/configuration#builder-options) guide. |

</div>

## Supported Next.js features

### `next/image`

`next/image` renders inside stories. The framework swaps Next.js's default image loader for a client-side mock, so stories don't depend on a `/_next/image` optimization endpoint — both local and remote `src` values render directly.

```tsx twoslash title="Image.stories.tsx"
// @noErrors
import Image from 'next/image'
import type { Meta, StoryObj } from 'storybook-next-rsbuild'

const meta = {
  component: Image,
  args: { src: '/vercel.svg', alt: 'Vercel', width: 200, height: 48 },
} satisfies Meta<typeof Image>

export default meta

export const Default: StoryObj<typeof meta> = {}
```

### `next/font`

Both `next/font/google` and `next/font/local` work out of the box — the synthetic `target.css` that `next-swc` emits is routed to a dedicated font loader (ported from `@storybook/nextjs`) that injects the `@font-face`/class CSS at runtime. CSS variables via the `variable` option also work. This is the one CSS concern the framework handles itself; everything else stays on Rsbuild's pipeline.

```tsx twoslash title="Font.stories.tsx"
// @noErrors
import { Inter } from 'next/font/google'
import type { Meta, StoryObj } from 'storybook-next-rsbuild'

const inter = Inter({ subsets: ['latin'] })

function FontDemo({ className }: { className: string }) {
  return <p className={className}>The quick brown fox jumps over the lazy dog.</p>
}

const meta = { component: FontDemo } satisfies Meta<typeof FontDemo>
export default meta

export const Default: StoryObj<typeof meta> = {
  args: { className: inter.className },
}
```

### `next/link`, `next/head`, `next/dynamic`

Ship unchanged. `next/head` updates `document.head` through a built-in decorator. `next/link` falls back to client-side navigation through the mocked router (see below).

### App Router and Pages Router

Use `parameters.nextjs` to choose between the two routers and pre-seed their state:

```tsx twoslash title="Navigation.stories.tsx"
// @noErrors
import { useRouter, usePathname, useSearchParams } from 'next/navigation'
import type { Meta, StoryObj } from 'storybook-next-rsbuild'

function Component() {
  const router = useRouter()
  const pathname = usePathname()
  const searchParams = useSearchParams()

  return (
    <button type="button" onClick={() => router.push('/next')}>
      Navigate from {pathname}?{searchParams.toString()}
    </button>
  )
}

export default {
  component: Component,
  parameters: {
    nextjs: {
      appDirectory: true,
      navigation: {
        pathname: '/hello',
        query: { foo: 'bar' },
      },
    },
  },
} satisfies Meta<typeof Component>

export const Default: StoryObj<typeof Component> = {}
```

- `appDirectory: true` selects the App Router (`next/navigation` hooks).
- `appDirectory: false` (default) selects the Pages Router (`next/router` hooks).
- `navigation` applies to the App Router; `router` applies to the Pages Router. Both accept a partial `NextRouter` shape.

### CSS, CSS Modules, PostCSS/Tailwind, `styled-jsx`

**Rsbuild owns the CSS pipeline** (this is the deliberate division of labor described in the [mental model](#configuration-mental-model)). CSS Modules, global CSS imports, PostCSS, and Tailwind work out of the box with no extra wiring — Rsbuild already handles them, and they behave the same as in your Next.js app. `styled-jsx` works as well; it is an SWC transform, so it flows through Next.js's loader chain.

### Sass / Less

Because Rsbuild — not Next.js — runs the CSS pipeline, Sass and Less are opt-in through an Rsbuild plugin rather than Next.js's built-in preprocessor support. Install the plugin and merge it via `rsbuildFinal`:

<PackageManagerTabs command="install @rsbuild/plugin-sass -D" />

```ts twoslash title=".storybook/main.ts"
// @noErrors
import { mergeRsbuildConfig } from '@rsbuild/core'
import { pluginSass } from '@rsbuild/plugin-sass'
import type { StorybookConfig } from 'storybook-next-rsbuild'

const config: StorybookConfig = {
  framework: 'storybook-next-rsbuild',
  stories: ['../src/**/*.stories.@(js|jsx|mjs|ts|tsx)'],
  // Rsbuild owns the CSS pipeline — extend it the Rsbuild way. The plugin
  // merges into the config the framework's bridge produces.
  rsbuildFinal: (config) =>
    mergeRsbuildConfig(config, { plugins: [pluginSass()] }),
}

export default config
```

Pick the `@rsbuild/plugin-sass` version compatible with the `@rsbuild/core` your [version matrix](#version-matrix) row pins.

### SWC transforms, `transpilePackages`, `optimizePackageImports`

Stories are compiled with Next.js's own `next-swc-loader`, so build behavior matches your app:

- `'use client'` directives behave like they do in Next.js.
- `server-only` modules resolve as in a real build.
- `transpilePackages` entries in your `next.config.ts` apply to stories automatically.
- `optimizePackageImports` (default-on in Next 15+) is honored — the `__barrel_optimize__` requests `next-swc` emits are routed through the SWC chain, so even TypeScript-source barrels compile correctly.
- JSX runtime selection follows your `next.config.ts`.

### CSS-in-JS via the SWC compiler (styled-components, Emotion)

These are Next.js compiler transforms, so you enable them the Next.js way — in `next.config.ts` — and they flow through the bridged `next-swc` chain. No Storybook-side wiring:

```ts twoslash title="next.config.ts"
// @noErrors
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  compiler: {
    styledComponents: true,
  },
}

export default nextConfig
```

### Advanced: custom loaders that span both layers (e.g. SVGR)

A few setups need to touch **both** config layers because Next.js's webpack config and Storybook's preview config don't see each other. SVGR is the canonical case: you add the loader rule the Next.js way (so it's part of the bridged config), but you also have to take `.svg` away from Rsbuild's default asset rule — a preview-side concern — via `webpackFinal`. The framework runs your `webpackFinal` **late, against the fully assembled rspack config**, so introspecting and mutating existing rules works as expected:

```ts twoslash title="next.config.ts"
// @noErrors
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  webpack: (config) => {
    config.module?.rules?.push({ test: /\.svg$/, use: ['@svgr/webpack'] })
    return config
  },
}

export default nextConfig
```

```ts twoslash title=".storybook/main.ts"
// @noErrors
import type { StorybookConfig } from 'storybook-next-rsbuild'

const config: StorybookConfig = {
  framework: 'storybook-next-rsbuild',
  stories: ['../src/**/*.stories.@(js|jsx|mjs|ts|tsx)'],
  // Exclude .svg from Rsbuild's default asset rule so the SVGR rule added in
  // next.config.webpack() is the one that processes them.
  webpackFinal: async (config) => {
    for (const rule of config.module?.rules ?? []) {
      if (
        rule &&
        typeof rule === 'object' &&
        rule.test instanceof RegExp &&
        rule.test.test('probe.svg')
      ) {
        ;(rule as any).exclude = /\.svg$/
      }
    }
    return config
  },
}

export default config
```

## Mocking Next.js APIs in stories

For interaction testing and manual overrides, the framework ships subpath exports that mirror `@storybook/nextjs-vite`. Each entry re-exports the real Next.js module and wraps select APIs with spy-able [`fn()`](https://storybook.js.org/docs/writing-tests/interaction-testing) mocks from `storybook/test`.

<div style={{ width: '100%', maxWidth: '800px' }}>

| Import | Use for |
| :--- | :--- |
| `storybook-next-rsbuild/navigation.mock` | App Router — `useRouter`, `usePathname`, `useSearchParams`, `redirect`, `notFound`, `useParams`, … |
| `storybook-next-rsbuild/router.mock` | Pages Router — `useRouter`, `withRouter`, and the singleton router |
| `storybook-next-rsbuild/cache.mock` | `revalidatePath`, `revalidateTag`, `unstable_cache`, `unstable_noStore` |
| `storybook-next-rsbuild/headers.mock` | `headers`, `cookies`, `draftMode` |

</div>

Call `getRouter()` from inside a `play` function to assert router interactions:

```tsx twoslash title="Navigation.stories.tsx"
// @noErrors
import { expect, userEvent, within } from 'storybook/test'
import { getRouter } from 'storybook-next-rsbuild/navigation.mock'
import type { StoryObj } from 'storybook-next-rsbuild'

export const Default: StoryObj = {
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement)
    const router = getRouter()

    await userEvent.click(canvas.getByRole('button'))
    await expect(router.push).toHaveBeenCalledWith('/next')
  },
}
```

:::warning Caveats
- **Singleton state.** `getRouter()` returns the instance seeded by the most recent story render. Don't hold on to the reference across stories — read it fresh inside each `play`.
- **Client-side only.** Storybook does not run a Next.js server, so `cache.mock` and `headers.mock` are the only way server-only APIs resolve when imported from client components in stories.
- **Coupled to Next.js internals.** These subpaths re-export through `next/dist/*` paths. A future `next` release may rename them, which typically requires a `storybook-next-rsbuild` upgrade.
:::

## Known limitations

- **No Server Components runtime.** Components marked `'use client'` render; pure Server Components are not executed. This matches `@storybook/nextjs-vite`.
- **No API routes, middleware, or server actions.** Storybook doesn't run a Next.js server — `route.ts`, `middleware.ts`, and `'use server'` entry points don't execute.
- **No `/_next/image` optimization.** `next/image` uses a client-side mock loader; runtime behavior differs from production where the image is optimized on the fly.
- **Sass/Less need an Rsbuild plugin.** Because Rsbuild owns the CSS pipeline, preprocessor support is opt-in via `rsbuildFinal` rather than inherited from Next.js (see [Sass / Less](#sass--less)).
- **Version coupling.** The bridge depends on Next.js internal modules. Any `next` patch or minor release can break compatibility — upgrade `storybook-next-rsbuild` alongside `next`.
